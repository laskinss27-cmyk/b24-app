# Вынос склада из Б24 в ERPNext («покрывало»)

## Зачем

Складской учёт внутри облачного Б24 упирается в фундамент: REST не отдаёт и не принимает ключевые вещи (стены — см. b24-rest-grabli.md), каждая мелкая цель требует огорода обходов. Решение руководства (2026-06-11): склад выносится в отдельное ядро. Б24 остаётся CRM.

## Архитектура «покрывало» (видение Сергея, зафиксировано 2026-06-11)

- **ERPNext — headless-движок.** Никто в нём не работает и даже его не видит. Он считает остатки, хранит документы и историю.
- **Все складские документы** (реализации, приходы, списания, перемещения, инвентаризации) — **чистые ERPNext-сущности**, в Б24 их нет. Связь со сделкой — поле `b24_deal_id` в документе (читается/пишется свободно — то, ради чего в Б24 были стены).
- **Люди работают только в нашем приложении внутри Б24** («покрывало»): у менеджеров, снабжения, инвентаризации — свои кнопки, которые через наш бэкенд управляют ERPNext. Робот закупок Владимира заменяется нашим механизмом снабжения.
- **Сделка-карточка остаётся сущностью Б24** (CRM-ткань: стадии, задачи, видимость 25 менеджерам, отчёты руководства). Открытый вопрос: товарные строки сделки — выносить в ERPNext (Б24 несёт только сумму; нативные товарные отчёты слепнут) или держать тонкую проекцию. Решается с Владимиром.
- Текущие фичи на `sale.order`/`crm.orderentity` — переходный режим до переключения, потом legacy.

## План (две фазы)

1. **«Домашний» этап (~2 недели до официальных):** полностью рабочая система на ноутбуке Сергея, канарейкой только для него. Б24 — источник правды, двойной учёт.
2. **Официальные 2 недели (после возвращения Владимира, площадки от него):** переезд на выданную инфраструктуру (тот же идемпотентный скрипт миграции + перенос тома), переключение людей, допил.

## Скрипт миграции: `scripts/erp-migrate-catalog.ts`

Идемпотентный (повторный запуск догоняет дельту). Фазы: `--dry` (посчитать) / `--stores` / `--items` / `--stock` / `--check`. env: `ERPNEXT_URL`, `ERPNEXT_TOKEN`, `LOCAL_PROXY`. На новой площадке: прогнать фазы по порядку → `--check` до нуля.

## Скрипт миграции клиентов: `scripts/erp-migrate-clients.ts`

Идемпотентный, тот же каркас. Фазы: `--dry` / `--setup` (custom-поля `b24_id` + Customer Group «Клиенты Б24» + **нумерация Customer серией** `Selling Settings.cust_master_name='Naming Series'` — иначе тёзки падают дублем имени) / `--companies` / `--contacts` / `--check`. Грабли в нём:
- frappe запрещает `<>` в именах → имена чистим;
- кривые email валятся на `validate_email_address` → фильтр-сито + ретрай создания Contact без контактных данных;
- ERPNext-вызовы: сетевые сбои ретраим (3×, headersTimeout/bodyTimeout 60с), HTTP-ошибки — НЕТ (повторный POST после фактического 2xx = дубли);
- 🔴 **долгие прогоны НЕ запускать фоном из Claude-сессии** — на ноутбуке Сергея такие процессы дважды умирали молча через ~20–40 мин (без stderr, журнал Windows чист, причина не найдена). Запускать отвязанно: `Start-Process cmd '/c npx tsx scripts\erp-migrate-clients.ts --contacts > scripts\migrate-clients.log 2>&1 & echo EXITCODE %ERRORLEVEL% >> ...'` и следить за логом.

## Грабли ERPNext (выстраданы 2026-06-11)

1. **Две компании в инсталляции**: «Умный дом» (УД) и «Умный дом (Demo)» (УДD, демо-данные SKU005–010) — **дефолт ДЕМО!** Во все документы `company` явно. Демо-компанию снести при уборке (на чистой площадке ставить без demo).
2. **Submit через REST = `PUT /api/resource/<DT>/<name> {"docstatus":1}`** и проверить `docstatus` в ответе. `frappe.client.submit` с частичным doc возвращает 200, НЕ проведя документ.
3. **Opening Stock** требует `expense_account` типа Temporary («Temporary Opening - УД»), иначе 417.
4. Склады ERPNext именуются `<Название> - <Аббр компании>` — учитывать в маппинге.
5. **Сеть ноутбука**: включён VPN-клиент (socks `127.0.0.1:10808`, http `10809`) — напрямую Битрикс не отвечает, Node прокси не уважает, undici ProxyAgent давал необъяснимые таймауты → Б24 в скриптах читается **через `curl -x http://127.0.0.1:10809`**; ERPNext (localhost) — напрямую. На площадке без прокси работает как есть.
6. Docker Hub из РФ рвёт большие образы — повторы + зеркало `cr.yandex/mirror/<img>`.
7. ERPNext API: `Authorization: token <key>:<secret>` (ключ Administrator через bench console), список — `limit_page_length=0`, фильтры — `filters=[["f","=","v"]]`.
