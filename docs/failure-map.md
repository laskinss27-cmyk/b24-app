# Карта отказов и сухие точки

Написано в ответ на страх «слишком много механизмов, каждый винтик — точка отказа» (Сергей, 2026-06-11). Метод: посчитать винтики честно, у каждого записать режим отказа и деградацию.

## Что РЕАЛЬНО работает в проде (полный инвентарь)

Движущихся частей в продакшене (после ухода с Yandex Cloud 2026-06-16, своя инфра):

1. **Битрикс24** — был до нас, будет после нас. Не наш винтик.
2. **Спейр-ноут (домашний сервер)** — docker-контейнер `b24-backend` (бэкенд + фронт-бандл) + ядро ERPNext рядом. Падает → рестарт/откат за минуту (`docs/SOS.md`). Аптайм = аптайм ноута (вкл+интернет).
3. **VPS reg.ru + ssh-туннель + nginx** — публичная дверь, выставляет контейнер наружу для Б24. Туннель отвалился → дверь 502, лечится `systemctl restart b24-tunnel`.
4. **Хранилища entity** (`ctv_inv`, `ctv_repairs` и др.) внутри самого Б24 — надёжность = надёжность Битрикса.
5. **Данные ядра** (каталог/остатки) — в Docker-томах на спейре (`erpnext_db-data`, `erpnext_sites`).

> Глаза на всё это — команда **`status`** на спейре; восстановление по сценариям — **`docs/SOS.md`**.

**Скрипты (~50 штук в `scripts/`) — НЕ винтики.** Это строительные леса: разведки и миграции, запускаются руками, в проде не живут, отказать в проде не могут. Их количество — мера проделанной разведки, не сложности системы.

## Карта отказов (прод сегодня)

| Узел | Отказ | Что видит менеджер | Деградация | Починка |
|---|---|---|---|---|
| Контейнер b24-backend | упал/завис | наши вкладки не грузятся | **нативный Б24 работает как всегда** — наши экраны надстройка, не несущая стена | `docker restart b24-backend`, ~5 сек (SOS.md) |
| Спейр выключен/нет сети | приложение лежит у всех | то же — нативный Б24 цел | данные в томах целы, поднимется при включении | включить ноут → автозапуск; `status` |
| Туннель/VPS отвал | дверь снаружи 502 | вкладки не грузятся | — | `systemctl restart b24-tunnel` (SOS.md) |
| Деплой кривого образа | баг в проде | видит только канарейка (бета-гейт) | менеджеры на GA-версии/нативе | откат на старый sha, 1 команда (runbook/SOS) |
| BX24 SDK флапает | вызов завис | «загрузка…» пару сек | таймауты + фолбэки: пустой список вместо вечного спиннера; данные идут через бэкенд | само (ретраи) |
| Б24 REST моргнул | fetch failed | ошибка текстом в нашем UI | повтор руками («↻ Обновить») | само |
| Кэш устарел (5 мин TTL) | старые цифры | чуть несвежие остатки | `force`-обновление кнопкой | само |
| Пишущий роут упал на полпути | партия недосоздана | текст ошибки + список созданных ID | ничего не удаляем молча; артефакты видны и чинятся руками | по логам шагов |

Незаменимых узлов без деградации: **ноль**. Худший сценарий любого отказа — «менеджеры работают, как работали до нас».

## Карта отказов (после покрывала — проектируется ТАК, иначе не принимаем)

| Узел | Отказ | Деградация |
|---|---|---|
| ERPNext недоступен (ноутбук спит / туннель упал) | остатки не читаются | **фича-флаг автоматически падает на цифры Б24**; пишущие кнопки честно говорят «ядро недоступно, попробуй позже» — ничего не теряется, очередей-полуопераций не бывает |
| Синк Б24→ERPNext отстал | расхождения | ежедневная авто-сверка кричит ДО того, как заметят люди; источник правды в переходный период — Б24 |
| Том докера погиб | данные ядра | ежедневный бэкап + Б24 ещё живёт параллельно = пересоздаваемо скриптом миграции |

Правило проектирования: **новый механизм без строки в этой таблице не деплоится.**

## Сухие точки (стоп-критерии — когда признаём «не получилось» и спокойно вылезаем)

1. **Ни один инструмент не переключается на ERPNext**, пока авто-сверка Б24↔ERPNext не стоит в нуле **7 дней подряд** на живых операциях.
2. **Менеджеры не переключаются**, пока Сергей не прожил на ERPNext-цифрах 2 недели без единого «цифра наврала».
3. **Если к началу официальных двух недель** операционный слой не работает стабильно у Сергея — в них едем с текущим переходным режимом (sale.order-партии), и это **штатный, заранее одобренный исход**, а не провал: инструменты уже лучше вчерашних.
4. **Б24-учёт не выключается** (не замораживается даже) без письменного «да» Сергея + Владимира. До этого момента отступление в любой день стоит ноль.
5. Любое «давай быстрее, пропустим сверку» — красный флаг; этот документ старше любой спешки.

## Почему сложность сейчас НА ПИКЕ и дальше будет падать

Сегодня мы несём двойной груз: обходы стен Б24 (sale.order-акробатика, чтение склада через резервы, смарт-процессы снабжения) И строящееся ядро. Это пик. Покрывало existiert ровно затем, чтобы **выкинуть** слой обходов: вместо пяти REST-цепочек с гардами — поле `b24_deal_id` и два запроса. Конечная система **проще** сегодняшней, не сложнее. Легаси-карта того, что отомрёт — в features.md.
