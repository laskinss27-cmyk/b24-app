# Фичи и экраны

Все экраны — наш React-бандл в iframe Б24 (см. architecture.md). Доступ к новому — канарейка `BETA_USER_IDS`.

## База товаров (`ProductBase.tsx`)

Каталог-браузер склада: полный каталог (товары + вариации плоско), остатки по складам, поиск, фильтр по складу/разделу, «только с остатком», фото. Сборка каталога — на бэкенде (`/api/catalog/browse`, кэш TTL 5 мин, `force` — пересборка). Кнопки шапки: «Отчёт по продажам», «📄 Реализации», «Инвентаризация».

**Режим пикера** (`picker={onDone,onCancel,title}`): тот же каталог как страница выбора позиций с колонкой «Кол-во» и липкой панелью «✓ Готово (N)» — используется вкладкой сделки для «Добавить товар».

## Вкладка сделки «Товары сделки» (`DealProductsTab.tsx`)

Замена стандартной вкладки «Товары» (placement в карточке сделки). Только для беты, остальным заглушка.

- **Таблица строк**: товары (TYPE 1 и 4!) и работы (TYPE 7) раздельными блоками, цены/кол-во/суммы, прибыль (закупка из каталога; работы × коэф из `app.option profit_coef`). Основной источник товаров — план ядра (Sales Order); если товар есть только в строках Б24 (старая/ручная сделка), вкладка показывает его как фолбэк, чтобы менеджер не терял позицию.
- **Строки сделки читаются через бэкенд** (`/api/deal/shipped → rows`, серверный `crm.item.productrow.list`) — фронтовый BX24 флапает; BX24 — фолбэк.
- **Чипы остатков по складам** (справочно). Селектора склада НЕТ намеренно: склад реализации выбирается в карточке черновика, прочитать его обратно для проведённых нельзя (стена 2) — врущей «памяти» не держим.
- **Партии реализации** («расщепление строки», дизайн Сергея): реализованная часть застывает строкой-записью (кол-во · склад · чип документа `#N/M ✎черновик/✓проведена` — клик открывает карточку), под ней строка остатка («60 шт из 80») с полем кол-ва и кнопкой «Реализовать». Каждая партия = свой черновик отгрузки; один заказ на сделку (нативная модель `#N/2,/3…`).
  - Склад у записи-партии: черновик → живьём из резервов документа (зелёным) или «⚠ склад не выбран»; проведённая → «склад — в карточке».
  - Бэкенд-цикл `/api/deal/realize`: контроль остатков → (заказ сделки есть? переиспользуем : `sale.order.add` → снос свежего авто-дубля → `crm.orderentity.add`) → корзина `crm_pr_` с полным кол-вом строки → черновик `sale.shipment.add (deducted:N)` + `shipmentitem` с кол-вом партии. При ошибке на полпути НИЧЕГО не откатываем — возвращаем `created{...}` для ручной зачистки.
- **«↪ Переместить» со сплитом** (`TransferSplitModal`): на строке статуса transfer кнопка открывает модалку — недостача распределяется по нескольким складам-источникам (старт: 1 запись = склад с макс. остатком; «+ источник»; кол-во ≤ остаток источника; индикатор «распределено X/Y»). Каждый источник = отдельный документ перемещения (бэкенд `groups`: N групп → N документов + задачи снабжению). Закрытие круга: после «Получено» строка показывает «✓ получено — обновить» → `reload()` → остаток склада Б из ядра → строка «хватит» → «Реализовать».
- **Чекбоксы строк — универсальный выбор**: менеджер отмечает нужные позиции, а нижняя панель применяет к ним действие. «Реализация» берёт отмеченные доступные товары и создаёт черновики Delivery Note по складам; «Заказать» формирует документ снабжения (Material Request) по отмеченным товарам, который появляется в дисплее «Снаб».
- Перед созданием заказа снабжению открывается проверочное окно со всеми выбранными позициями, количеством и складом назначения. У каждой позиции можно оставить отдельный комментарий; он сохраняется в строке Material Request и показывается снабжению при раскрытии заявки.
- Позиция, уже включённая в открытую заявку снабжению, получает нейтральную метку «заказано» рядом с названием независимо от текущего наличия. Такая позиция исключается из повторного действия «Заказать», но её по-прежнему можно реализовать, если товар появился на выбранном складе.
- **«Добавить товар»**: открывает Базу в режиме пикера → пачкой `crm.item.productrow.add` (`/api/deal/add-products`); существующие строки не трогаются.
- **Экспорт сделки в Excel**: строки выгружаются двумя последовательными блоками — сначала товары, затем работы. Внизу файла выводятся отдельные суммы «Товары», «Работы» и общий «Итого».
- **Варианты КП**: до выбора клиента варианты можно создавать, копировать, переименовывать и редактировать. Кнопка выбора на выбранном варианте превращается в «Отменить выбор клиента»: текущий рабочий состав сохраняется обратно в вариант, после чего расчёты снова доступны для редактирования. Отмена и смена варианта блокируются, если по сделке уже есть этапы, заявки снабжению, реализации или перемещения. Внутреннее название варианта не передаётся в клиентское КП и Excel — варианты различаются по составу и итоговой сумме.
- **Заказ снабжению** уходит в ядро как `Material Request`: менеджер не выбирает склад-источник, а только отмечает позиции и формирует заказ; дальше снабженец разбирает его в дисплее «Снаб». Старый путь через смарт-процесс Б24 1110 остаётся legacy и не используется для кнопки «Заказать».

### Доработки вкладки сделки (2026-06-24)
- **Фрейм на всю высоту**: `BX24.fitWindow()` + ResizeObserver на body — Б24 держал ~половину страницы, появлялся внутренний скролл; теперь iframe растягивается под контент (и при подгрузке/раскрытии).
- **Инлайн-правка строки** (товары и работы): прямо в ячейках — **базовая цена + скидка % + кол-во**; итог «=N/ед» и сумма пересчитываются на лету, сохранение при уходе фокуса ИЗ строки (`onRowBlur` → `/api/deal/update-product`). У товара **два поля кол-ва**: «кол-во» (в сделке, сохраняется) и «отгруз.» (сколько реализовать сейчас — отдельная механика). Кнопки-модалки ✎ нет (была — убрана). **🗑 удаление** строки — `/api/deal/remove-product`. Оба пишут надёжным путём: `crm.deal.productrows.get` → правим/убираем → `crm.deal.productrows.set` (тот же id-простор). Заблокированы во время реализации.
- **Скидка % на позицию** (плавающая, менеджер сам): инлайн в строке (база цены + скидка % → итог). Б24-модель: PRICE=итог за ед., DISCOUNT_SUM=скидка/ед, DISCOUNT_RATE=%. После переезда склада подбор+цены+скидки целиком в нашем окне (нативная вкладка «Товары» не нужна).
- **Закупка в строке товара**: под розницей «закуп N ₽»; красным с ⚠, если розница ≤ закупки (продажа по закупу/в минус); «закуп —» серым, если не заполнена. Из `purchasingPrice` (ядро/каталог).
- **Оплата из «Кассы»** (исправлен баг «не оплачено» на оплаченных сделках): оплату ведёт смарт-процесс «Касса» в полях сделки `UF_CRM_1765984372` («Сумма оплат») / `UF_CRM_1765984397` («Остаток к оплате»), а НЕ платежи заказа (в заказе их 0). Баннер берёт оплату оттуда (фолбэк на платежи заказа). `KASSA_PAID_FIELD`/`KASSA_REMAIN_FIELD` в api-deal.
- **Перемещение у менеджера полностью убрано** (2026-06-24): кнопка «↪ Переместить» и затем бейдж «нужно перемещение» — оба сняты (в статусе transfer у менеджера пусто; активное/полученное перемещение от снабжения показывается как статус). Менеджер логистику не инициирует. Оформление перемещений уйдёт снабжению (отдельная часть). `TransferSplitModal` оставлен в коде под переиспользование.
- **Остатки по складам — свёрнуты** (2026-06-24): показываем 2 склада (по убыванию остатка) + «+N»; при наведении на строку раскрываются все (CSS `nth-child(n+3)` + `tr:hover`) — чтобы длинные списки складов не раздували строки.

## Дисплей снабжения (`Supply.tsx`, 2026-07-12)

- Главный экран «Обеспечение и заказы» показывает все заявки `Material Request` из ядра; заявка раскрывается в запрошенные позиции и связанные документы.
- Одну позицию можно распределить на несколько источников: несколько складов, закупка и перемещение одновременно, несколько поставщиков. Одинаковые склады объединяются в одно перемещение, одинаковые поставщики — в одну заявку поставщику.
- Перемещение ограничено остатком выбранного склада и незакрытой потребностью заявки. Закупка может быть больше потребности: излишек считается запасом и не увеличивает потребность исходной заявки.
- Перед записью показывается сводка создаваемых документов. Перемещения создаются с привязкой к заявке/сделке и сразу проводятся на транзитный склад; заявки поставщику создаются черновиками для последующего заполнения цен и согласования.
- Сервер повторно сверяет заявку, уже созданные документы и живые остатки; параллельное или повторное создание по устаревшему экрану блокируется. При частичном сбое ответ перечисляет документы, которые уже успели создаться.
- Дочерние документы показаны компактным деревом под заявкой. Статусы нейтральные и отражают только состояние документа: «Черновик», «На согласовании», «Заказано», «В пути», «Получено», «Недовоз», «Отменено».
- Номер документа кликабелен в заявке, дереве, реестрах закупок и логистики. Внутренняя карточка заявки поставщику показывает позиции, количество, цены, сумму и оприходования; карточка перемещения — маршрут, отправленное, полученное и недовоз.
- Карточка заявки поставщику рабочая: можно изменить поставщика, количество, закупочную цену, удалить строку и перевести документ между этапами «Черновик» → «На согласовании» → «Согласовано» → «Заказано» (также доступна отмена). В карточке перемещения доступны только допустимые складские переходы: «В путь», фактический приём с количеством и завершение недовоза.
- После этапа «Заказано» в карточке закупки появляется фактическая приёмка по каждой позиции. Разрешены несколько частичных оприходований; сервер не даёт принять больше заказанного и всегда направляет приход на `SUPPLY_RECEIPT_STORE` (по умолчанию «Склад Прихода»), а не сразу на склад точки.
- Сергей (пользователь `1858`) может удалить заявку поставщику или перемещение прямо из карточки документа. Сервер повторно проверяет пользователя; оприходования и складские проводки отменяются в обратном порядке, после чего документ исчезает из дисплея. Проведённые записи ядра сохраняются отменёнными для аудита, синхронизация остатков не меняется.
- Имена складов перед проводкой нормализуются идемпотентно: как короткое имя Б24, так и уже квалифицированное ERPNext-имя `Склад - УД` превращаются ровно в одно ERPNext-имя. Это защищает проводки от ошибочного суффикса `- УД - УД`.
- **Печатные формы закупки**: из карточки заявки поставщику печатается чистая заявка с текущими позициями, количеством, ценами, суммой и ожидаемой датой (в том числе с ещё не сохранёнными правками); системная печать позволяет сразу сохранить PDF для отправки поставщику. В исходной заявке снабжения доступна сводная форма на согласование по всем активным поставщикам: статусы, подытоги по поставщикам, общий итог и явная пометка одинакового товара у нескольких поставщиков.
- **Все складские документы в дисплее снабжения**: «Логистика» показывает полный журнал перемещений, включая самостоятельные и созданные по заявкам/закупкам. В левом меню также есть «Списания», «Оприходования», «Реализации» и «Возвраты». Строки кликабельны, открывают карточку с составом; номер сделки и ответственный отображаются в списке и карточке, если документ связан со сделкой.
- **Перенаправление перемещения**: снабжение может изменить склад назначения в карточке документа в статусах «Запрошено» и «В пути». Уже выполненная проводка на транзит не отменяется; новый склад используется при приёмке, изменение фиксируется в истории документа. После приёмки поле блокируется.

## Инвентаризация (`InventoryHome.tsx`, `InventoryReport.tsx`, `MobileCount.tsx`)

Инициаторы — `app.option inv_initiators`. **Мобильный QR**: кнопка «QR» у точки → телефон сканирует → `/m` (автономный OAuth вне iframe) → мобильный подсчёт → факты пишутся токеном телефона.

**Документ ядра (ERPNext, 2026-06-12):** у сверённой точки кнопка «Документ ядра» — 1С-цепочка болванка → «Записать» (черновик Stock Reconciliation) → «Провести» (остатки ядра + черновики-зеркала D/S в Б24). Роуты `/api/inventory/erp-doc-{preview,save,submit}`, гейт — env `ERPNEXT_URL` (нет — фича выключена). Ссылка на документ — `pt.erpDoc` в entity. Детали и грабли: docs/sklad-vynos.md.

## Окно «Складской учёт» (`StockLedger.tsx`)

Левое меню (view='stock', placement `/placement/stock`), под канарейкой (`BETA_USER_IDS`/админ). Вкладки: **Перемещения** / **Списания** / **Оприходования** / **Реализации**. На каждой — панель фильтров (поиск+статус мгновенно на клиенте, период с/по серверно в ядро). Журнал движений read-only — `/api/stock/movements` (`listCoreMovements`), перемещения — `/api/transfers/*`. У каждого документа — кликабельная ссылка на сделку + ФИО ответственного (всегда ASSIGNED_BY_ID сделки).

**Три формы создания (2026-06-22), право = канарейка `STOCK_CREATE_IDS={1,986,1858}` (гейт и на сервере):**
- **Приход** (вкладка Оприходования): склад + поставщик + товары · кол-во · закупка · розница → черновик `Purchase Receipt` → «Провести». Закупка → valuation ядра; розница (если заполнена) → best-effort пуш в Б24 (`catalog.price`, тип розница=2). Розница «как мастер в ядре» — НЕ сделано (нет потребителя + price-list не выверен), следующий шаг.
- В заявке поставщику после фактического прихода можно явно создать перемещение со Склада Прихода на склад точки. В транзит уходит только оприходованное количество в пределах незакрытой потребности заявки; закупленный запас остаётся на приходном складе.
- **Поставщик** = пикер-поиск по CRM-контрагентам Б24 (`/api/stock/search-contractors` → `crm.company.list %TITLE`; складские контрагенты Б24 приходят из CRM, отдельного `catalog.contractor` REST нет). Выбор из подсказок или ввод нового → имя заводится в ядре (`ensureSupplier`). Пусто → тех. «Б24 Снабжение».
- **Списание**: склад + причина (custom-поле `b24_reason` на Stock Entry) + товары · кол-во → черновик `Material Issue` → «Провести».
- **Перемещение** (выбор Сергея — честный транзит, НЕ прямой A→B): склад откуда/куда + товары · кол-во → `/api/transfers/create-manual` → `ctv_transfer` без сделки в статусе «Запрошено»; дальше штатные «В пути»/«Получено» проводит снабжение (двухфазный транзит через `Goods In Transit`).
- **Примечание** (необязательное) во всех трёх формах: приход/списание — custom-поле `b24_note` на документе ядра (видно в журнале, колонка «Инфо»); перемещение — поле `note` в `ctv_transfer` (видно под маршрутом).
- **Добавление позиций — логика 1С** (`AddItemModal`): в окне создания список пуст, кнопка «➕ Добавить товар» открывает под-форму (поиск `searchStockItems` по каталогу ядра → выбор → кол-во, +закупка/розница для прихода → «Добавить»), позиция падает в список; для следующей — снова «Добавить товар». Повтор того же товара суммирует кол-во. Строки списка редактируются (кол-во/цены) + удаление.
- **Остатки в пикере товара** (все пикеры — приход/списание/перемещение, фильтры, отчёт): `/api/stock/search-items` обогащает результаты остатками по складам (батч `fetchErpStocksFor`). В строке результата — «Σ всего · топ-склады», «нет на складах» красным (живая карточка среди дублей видна по факту наличия). После выбора в под-форме — полная разбивка по складам; для списания/перемещения склад-источник (`highlightStore` = склад «откуда») подсвечен + предупреждение, если вводимое кол-во больше остатка.
- **Новый товар (нет в каталоге)** — в под-форме ссылка «Создать новый товар» → ввод названия → `/api/stock/create-product`: `catalog.product.add` (iblock 24 = базовый каталог CRM, type 1 простой, measure 9 штуки) → productId → `ensureCoreItem` (Item ядра, code=productId, group «Каталог Б24», uom «шт», stock-item) → товар сразу выбран, указываешь кол-во/цены. Архитектура сохранена: новый товар получает Б24 productId, консистентен с остатками/сделками/синком.
- Поставщики — `/api/stock/form-data` (Б24-воронка контрагентов, см. выше); склады + право — там же. Поведение: черновик → отдельная кнопка «Провести» (не авто-сабмит).
- **Открыть документ** — строки журнала (приход/списание/реализация) кликабельны → модалка с содержимым (`/api/stock/doc` → `erp.get` doctype+name): позиции (товар·кол-во·склад·цена), дата/статус, поставщик/причина/примечание, ссылка на сделку. Перемещения раскрываются клиентски (позиции + история статусов из `ctv_transfer`).
- **Поиск по товару в каждой вкладке** — пикер `ProductFilter` (поиск по каталогу ядра) фильтрует список документов вкладки по товару: для движений ядра — серверно по дочерней таблице (`/api/stock/movements` + productId → frappe-фильтр `[child_doctype, item_code]`), для перемещений — клиентски по `lines`.
- **Печатные формы** (перемещение/списание/приход) — кнопка «🖨 Печать» в карточке документа (`StockBlank` + `@media print`, как КП/ремонты: `body *{visibility:hidden}` → виден только `.stock-blank`). Перемещение → «Накладная на перемещение» (из `TransferDoc`, кол-во без цены); списание → «Акт о списании» (склад+причина); приход → «Приходная накладная» (поставщик + цена/сумма из закупки). Шапка: организация, №/дата, стороны, **Основание** (сделка #N / иначе примечание / иначе «внутреннее перемещение»), ответственный (из сделки, для ручного — создатель), подписи по типу (Отпустил-Получил / Комиссия-Утвердил / Сдал-Принял). Только presentation над уже загруженными данными (`TransferDoc`/`CoreDocDetail`), бэкенд не трогали.
- **Вкладка «Отчёт по движению товара»** — выбираешь товар → вся история движений по всем типам (`/api/stock/item-history` → **Stock Ledger Entry** ядра): дата, тип (оприходование/списание/перемещение/реализация/инвентаризация — Stock Entry дорезолвлен по `stock_entry_type`), ±кол-во, склад, документ (кликается в детали). Отвечает на «когда товар последний раз приходил/продавался/списывался/перемещался».

## Ремонты — заморозка, две цены, авто-сделка (`Repairs.tsx` / `api-repairs.ts`, 2026-06-24)

- **Заморозка карточки**: со статуса `received_office` («Принято в офисе») любые изменения — поля, цены, вид, **статус**, удаление — только снабжение+ (`canEditPrice`: Вова 1 / Сергей 1858 / Бекасов 986 + отдел Снабжение 10). Серверный гейт (403) в `update`/`set-pay`/`update-status`/`delete` по ТЕКУЩЕМУ статусу (`isLocked` = индекс ≥ `received_office`): перевести ТТ→офис обычный приёмщик ещё может, дальше всё фиксируется. Фронт зеркалит (гасит кнопки + 🔒-плашка).
- **Две цены** у платных: `cost` → «Цена ремонта СЦ» (берёт сервис-центр), новое `ourPrice` → «Наша цена» (берём с клиента). Обе видны всем, правит снабжение+.
- **Авто-сделка** (`syncRepairDeal`): при простановке «Наша цена» > 0 на платном ремонте с привязанным контактом-клиентом — `crm.deal.add` (TITLE «Платный ремонт №N», CONTACT_ID, OPPORTUNITY = наша цена) + `crm.deal.productrows.set` [{PRODUCT_NAME:'Платный ремонт №N', PRICE, QTY:1}]; `dealId` пишется в карточку (дубля нет), смена цены позже обновляет сумму/позицию. Без контакта сделка не создаётся (подсказка «привяжи клиента»). Ссылка на сделку — в карточке.
- **Воронка «Ремонты»** (`ensureRepairsDealCategory`): сделки ремонтов льются в отдельную воронку сделок (entityTypeId=2, резолв по имени «Ремонты», создаётся бэкендом если нет, кэш на процесс) — отдельно от продаж и без робота-переименователя общей воронки «Объекты» (он съедал TITLE → «{ID}_»). Если воронку получить/создать не вышло — best-effort, сделка уходит в общую.
- **Комментарий СЦ** — заполняет/правит только снабжение+ (серверный гейт в create/update; у менеджеров поле textarea неактивно).
- **Акт выдачи после ремонта** доступен у клиентского ремонта в статусах «Готово к выдаче» и «Выдано». Печатаются два экземпляра с датами приёма/готовности/выдачи, оборудованием, заявленной неисправностью, исходной комплектностью, выполненными работами из поля «Комментарий СЦ», видом и стоимостью ремонта, фиксацией проверки, замечаний и подписями. Дату передачи в сервис и сведения о деталях документ намеренно не показывает: этих достоверных данных в карточке нет.
- **Список ремонтов** — колонка цены показывает «Наша цена» (`ourPrice`), не цену СЦ.

## Отчёт по продажам (`SalesReport.tsx`)

Выигранные сделки (`STAGE_SEMANTIC_ID='S'`) за период по `CLOSEDATE`: воронка, источник, менеджер, суммы товаров/услуг, прибыльность, счётчик позиций без закупки. CSV (UTF-8 BOM, `;`). Известная особенность данных: сделки без товарных строк (сумма руками в OPPORTUNITY) дают нули по товарам — решено пока не обрабатывать.

## Легаси-карта (после выноса склада)

Партии через `sale.order`/`crm.orderentity`, чтение склада из резервов и старое снабжение на смарт-процессах — **переходный режим**. Целевая модель — «покрывало» (sklad-vynos.md): кнопки управляют ERPNext-документами, а слой обходов Б24 постепенно отмирает. Не вкладываться в легаси без причины.
