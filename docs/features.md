# Фичи и экраны

Все экраны — наш React-бандл в iframe Б24 (см. architecture.md). Доступ к новому — канарейка `BETA_USER_IDS`.

## База товаров (`ProductBase.tsx`)

Каталог-браузер склада: полный каталог (товары + вариации плоско), остатки по складам, поиск, фильтр по складу/разделу, «только с остатком», фото. Сборка каталога — на бэкенде (`/api/catalog/browse`, кэш TTL 5 мин, `force` — пересборка). Кнопки шапки: «Отчёт по продажам», «📄 Реализации», «Инвентаризация».

**Режим пикера** (`picker={onDone,onCancel,title}`): тот же каталог как страница выбора позиций с колонкой «Кол-во» и липкой панелью «✓ Готово (N)» — используется вкладкой сделки для «Добавить товар».

## Вкладка сделки «Товары сделки» (`DealProductsTab.tsx`)

Замена стандартной вкладки «Товары» (placement в карточке сделки). Только для беты, остальным заглушка.

- **Таблица строк**: товары (TYPE 1 и 4!) и работы (TYPE 7) раздельными блоками, цены/кол-во/суммы, прибыль (закупка из каталога; работы × коэф из `app.option profit_coef`).
- **Строки сделки читаются через бэкенд** (`/api/deal/shipped → rows`, серверный `crm.item.productrow.list`) — фронтовый BX24 флапает; BX24 — фолбэк.
- **Чипы остатков по складам** (справочно). Селектора склада НЕТ намеренно: склад реализации выбирается в карточке черновика, прочитать его обратно для проведённых нельзя (стена 2) — врущей «памяти» не держим.
- **Партии реализации** («расщепление строки», дизайн Сергея): реализованная часть застывает строкой-записью (кол-во · склад · чип документа `#N/M ✎черновик/✓проведена` — клик открывает карточку), под ней строка остатка («60 шт из 80») с полем кол-ва и кнопкой «Реализовать». Каждая партия = свой черновик отгрузки; один заказ на сделку (нативная модель `#N/2,/3…`).
  - Склад у записи-партии: черновик → живьём из резервов документа (зелёным) или «⚠ склад не выбран»; проведённая → «склад — в карточке».
  - Бэкенд-цикл `/api/deal/realize`: контроль остатков → (заказ сделки есть? переиспользуем : `sale.order.add` → снос свежего авто-дубля → `crm.orderentity.add`) → корзина `crm_pr_` с полным кол-вом строки → черновик `sale.shipment.add (deducted:N)` + `shipmentitem` с кол-вом партии. При ошибке на полпути НИЧЕГО не откатываем — возвращаем `created{...}` для ручной зачистки.
- **«↪ Переместить» со сплитом** (`TransferSplitModal`): на строке статуса transfer кнопка открывает модалку — недостача распределяется по нескольким складам-источникам (старт: 1 запись = склад с макс. остатком; «+ источник»; кол-во ≤ остаток источника; индикатор «распределено X/Y»). Каждый источник = отдельный документ перемещения (бэкенд `groups`: N групп → N документов + задачи снабжению). Закрытие круга: после «Получено» строка показывает «✓ получено — обновить» → `reload()` → остаток склада Б из ядра → строка «хватит» → «Реализовать».
- **«Добавить товар»**: открывает Базу в режиме пикера → пачкой `crm.item.productrow.add` (`/api/deal/add-products`); существующие строки не трогаются.
- **«→ в снабжение»** у товаров «нет на складах»: создаёт карточку «Снабжение» («Поставка № N_<сделка>_<название>») с точным перечнем «имя × кол-во» или дополняет открытую заявку сделки; опционально «куда привезти» (поле «Склад поставки» заявки, если справочник iblock 60 читается токеном; всегда дублем строкой в перечень). Чипы заявок со стадиями внизу. Свои «Товарные позиции» НЕ создаём (конвейер робота — см. грабли).

## Окно «Реализации» (`Realizations.tsx`)

Зеркало нативного списка реализаций + колонка **«Сделка»** (кликабельные чипы). Сборка на бэкенде (`/api/realizations/list`, кэш 5 мин): `sale.shipment.list (deducted Y, system N, кап 200)` → заказы → разворот `crm_pr_` → сделки. Клиент — из свойств заказа. Колонки складов НЕТ (стена 2). Фильтры: период, поиск, «только проведённые».

## Инвентаризация (`InventoryHome.tsx`, `InventoryReport.tsx`, `MobileCount.tsx`)

Точки инвентаризации по складам/разделам, подсчёт фактов, акт расхождений. Хранилище — entity `ctv_inv` (через бэкенд `/api/inventory/*`; фронтовый BX24 на entity виснет). Инициаторы — `app.option inv_initiators`. **Мобильный QR**: кнопка «QR» у точки → телефон сканирует → `/m` (автономный OAuth вне iframe) → мобильный подсчёт → факты пишутся токеном телефона.

**Документ ядра (ERPNext, 2026-06-12):** у сверённой точки кнопка «Документ ядра» — 1С-цепочка болванка → «Записать» (черновик Stock Reconciliation) → «Провести» (остатки ядра + черновики-зеркала D/S в Б24). Роуты `/api/inventory/erp-doc-{preview,save,submit}`, гейт — env `ERPNEXT_URL` (нет — фича выключена). Ссылка на документ — `pt.erpDoc` в entity. Детали и грабли: docs/sklad-vynos.md.

## Окно «Складской учёт» (`StockLedger.tsx`)

Левое меню (view='stock', placement `/placement/stock`), под канарейкой (`BETA_USER_IDS`/админ). Вкладки: **Перемещения** / **Списания** / **Оприходования** / **Реализации**. На каждой — панель фильтров (поиск+статус мгновенно на клиенте, период с/по серверно в ядро). Журнал движений read-only — `/api/stock/movements` (`listCoreMovements`), перемещения — `/api/transfers/*`. У каждого документа — кликабельная ссылка на сделку + ФИО ответственного (всегда ASSIGNED_BY_ID сделки).

**Три формы создания (2026-06-22), право = канарейка `STOCK_CREATE_IDS={1,986,1858}` (гейт и на сервере):**
- **Приход** (вкладка Оприходования): склад + поставщик + товары · кол-во · закупка · розница → черновик `Purchase Receipt` → «Провести». Закупка → valuation ядра; розница (если заполнена) → best-effort пуш в Б24 (`catalog.price`, тип розница=2). Розница «как мастер в ядре» — НЕ сделано (нет потребителя + price-list не выверен), следующий шаг.
- **Поставщик** = пикер-поиск по CRM-контрагентам Б24 (`/api/stock/search-contractors` → `crm.company.list %TITLE`; складские контрагенты Б24 приходят из CRM, отдельного `catalog.contractor` REST нет). Выбор из подсказок или ввод нового → имя заводится в ядре (`ensureSupplier`). Пусто → тех. «Б24 Снабжение».
- **Списание**: склад + причина (custom-поле `b24_reason` на Stock Entry) + товары · кол-во → черновик `Material Issue` → «Провести».
- **Перемещение** (выбор Сергея — честный транзит, НЕ прямой A→B): склад откуда/куда + товары · кол-во → `/api/transfers/create-manual` → `ctv_transfer` без сделки в статусе «Запрошено»; дальше штатные «В пути»/«Получено» проводит снабжение (двухфазный транзит через `Goods In Transit`).
- **Примечание** (необязательное) во всех трёх формах: приход/списание — custom-поле `b24_note` на документе ядра (видно в журнале, колонка «Инфо»); перемещение — поле `note` в `ctv_transfer` (видно под маршрутом).
- **Добавление позиций — логика 1С** (`AddItemModal`): в окне создания список пуст, кнопка «➕ Добавить товар» открывает под-форму (поиск `searchStockItems` по каталогу ядра → выбор → кол-во, +закупка/розница для прихода → «Добавить»), позиция падает в список; для следующей — снова «Добавить товар». Повтор того же товара суммирует кол-во. Строки списка редактируются (кол-во/цены) + удаление.
- **Остатки в пикере товара** (все пикеры — приход/списание/перемещение, фильтры, отчёт): `/api/stock/search-items` обогащает результаты остатками по складам (батч `fetchErpStocksFor`). В строке результата — «Σ всего · топ-склады», «нет на складах» красным (живая карточка среди дублей видна по факту наличия). После выбора в под-форме — полная разбивка по складам; для списания/перемещения склад-источник (`highlightStore` = склад «откуда») подсвечен + предупреждение, если вводимое кол-во больше остатка.
- **Новый товар (нет в каталоге)** — в под-форме ссылка «Создать новый товар» → ввод названия → `/api/stock/create-product`: `catalog.product.add` (iblock 24 = базовый каталог CRM, type 1 простой, measure 9 штуки) → productId → `ensureCoreItem` (Item ядра, code=productId, group «Каталог Б24», uom «шт», stock-item) → товар сразу выбран, указываешь кол-во/цены. Архитектура сохранена: новый товар получает Б24 productId, консистентен с остатками/сделками/синком.
- Поставщики — `/api/stock/form-data` (Б24-воронка контрагентов, см. выше); склады + право — там же. Поведение: черновик → отдельная кнопка «Провести» (не авто-сабмит).
- **Открыть документ** — строки журнала (приход/списание/реализация) кликабельны → модалка с содержимым (`/api/stock/doc` → `erp.get` doctype+name): позиции (товар·кол-во·склад·цена), дата/статус, поставщик/причина/примечание, ссылка на сделку. Перемещения раскрываются клиентски (позиции + история статусов из `ctv_transfer`).
- **Поиск по товару в каждой вкладке** — пикер `ProductFilter` (поиск по каталогу ядра) фильтрует список документов вкладки по товару: для движений ядра — серверно по дочерней таблице (`/api/stock/movements` + productId → frappe-фильтр `[child_doctype, item_code]`), для перемещений — клиентски по `lines`.
- **Печатные формы** (перемещение/списание/приход) — кнопка «🖨 Печать» в карточке документа (`StockBlank` + `@media print`, как КП/ремонты: `body *{visibility:hidden}` → виден только `.stock-blank`). Перемещение → «Накладная на перемещение» (из `TransferDoc`, кол-во без цены); списание → «Акт о списании» (склад+причина); приход → «Приходная накладная» (поставщик + цена/сумма из закупки). Шапка: организация, №/дата, стороны, **Основание** (сделка #N / иначе примечание / иначе «внутреннее перемещение»), ответственный (из сделки, для ручного — создатель), подписи по типу (Отпустил-Получил / Комиссия-Утвердил / Сдал-Принял). Только presentation над уже загруженными данными (`TransferDoc`/`CoreDocDetail`), бэкенд не трогали.
- **Вкладка «Отчёт по движению товара»** — выбираешь товар → вся история движений по всем типам (`/api/stock/item-history` → **Stock Ledger Entry** ядра): дата, тип (оприходование/списание/перемещение/реализация/инвентаризация — Stock Entry дорезолвлен по `stock_entry_type`), ±кол-во, склад, документ (кликается в детали). Отвечает на «когда товар последний раз приходил/продавался/списывался/перемещался».

## Отчёт по продажам (`SalesReport.tsx`)

Выигранные сделки (`STAGE_SEMANTIC_ID='S'`) за период по `CLOSEDATE`: воронка, источник, менеджер, суммы товаров/услуг, прибыльность, счётчик позиций без закупки. CSV (UTF-8 BOM, `;`). Известная особенность данных: сделки без товарных строк (сумма руками в OPPORTUNITY) дают нули по товарам — решено пока не обрабатывать.

## Легаси-карта (после выноса склада)

Партии через `sale.order`/`crm.orderentity`, «→ в снабжение» на смарт-процессах, чтение склада из резервов — **переходный режим**: работает, пока складские документы живут в Б24. Целевая модель — «покрывало» (sklad-vynos.md): те же кнопки начнут управлять ERPNext-документами, и весь этот слой обходов отомрёт. Не вкладываться в его развитие без причины.
