# Архитектура

## Монорепо

```
packages/
  backend/   — Fastify-сервер: OAuth, placement-роуты, /api/* (серверные походы в Б24 REST)
  frontend/  — React SPA, один бандл; Битрикс показывает его в iframe через placement
  shared/    — общие TS-типы
scripts/     — разведки/тесты/миграции (untracked, см. scripts.md)
docs/        — ты здесь
```

Сборка: `npm run typecheck` → `npm run build` (оба прогоняют все workspaces). Бандл фронта кладётся в `packages/frontend/dist` и раздаётся бэкендом.

## Один бандл — несколько входов

`frontend/src/main.tsx` выбирает экран по контексту (`b24-context.ts`):
- `dealId` из placement-контекста → **DealProductsTab** (вкладка сделки);
- `view='inventory'` → **ProductBase** (левое меню: База товаров / инвентаризация — внутри сам решает по канарейке);
- `view='salesReport'` → отчёт по продажам;
- `view='mobileCount'` (`/m`, телефон вне iframe) → мобильный подсчёт инвентаризации;
- режимы `report` / `realizations` — внутри ProductBase (кнопки в шапке).

## Авторизация — два пути

1. **Фронт → наш бэкенд → Б24 (основной путь).** Фронт берёт токен юзера `BX24.getAuth()` и шлёт в наши `/api/*` (`{domain, accessToken}`). Бэкенд проверяет домен по allowlist (`normalizeDomain(cfg.portalDomain)`) и ходит в Б24 серверным `B24Client` **токеном юзера** — права Битрикса соблюдаются автоматически. Мобильный режим (`/m`): токен добывается автономным OAuth (см. грабли в b24-rest-grabli.md → OAuth) и живёт в шифрованной cookie.
2. **Вебхук `DEV_WEBHOOK` (только dev/скрипты).** Полный scope, для разведок и миграций. В прод-коде не используется.

`B24Client` (`backend/src/b24/client.ts`): throttle 8 rps (лимит Б24 — 10), `callBatch` группирует до 50 вызовов, ошибки пробрасываются как `B24ApiError` с кодом/описанием.

## Канарейка (бета-гейт)

`BETA_USER_IDS = ['1858' (Сергей), '986' (Бекасов), '1' (Дранишников)]` в `frontend/src/b24.ts` + `BX24.isAdmin()`. Новые фичи видят только они; остальным — заглушка «Раздел в разработке» или GA-версия. Это позволяет катать на прод смело: прод ≠ виден всем.

## Почему данные ходят через бэкенд, а не BX24 на фронте

Фронтовый BX24 SDK **флапает**: `app.option.get`, `crm.deal.productrows.get` на свежих сделках, любые вызовы при возврате из нативных окон — могут зависнуть навсегда. Правила:
- тяжёлое/важное — серверным клиентом через `/api/*` (чистый JSON, стабильно);
- что осталось на BX24 — только через `withTimeout` (+ `withRetry` для инициализации);
- у каждой загрузки мягкий фолбэк (пустой список лучше вечного спиннера).

## Хранилища приложения (entity)

Б24 `entity.*` — наше key-value на портале (работает только из app-контекста; `entity.add` — админская операция, создаётся бэкендом идемпотентно, `placement.ts`):
- `ctv_inv` — инвентаризации;
- `ctv_realize` — память складов партий (сейчас спит: от показа отказались, см. features.md → Партии).
