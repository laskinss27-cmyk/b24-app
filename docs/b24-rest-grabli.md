# Энциклопедия граблей и стен Битрикс24 REST (облако)

Всё ниже выстрадано живыми тестами на портале `umniydom.bitrix24.ru` (облако: БД/исходников нет, только REST). Даты — когда проверено. **Главная мета-грабля: список `methods` НЕ полон — существование метода проверяется только живым вызовом** (валидационная ошибка = метод есть; `ERROR_METHOD_NOT_FOUND` = нет).

## Стена 1 (ПРОБИТА 2026-06-11): привязка заказ ↔ сделка

Карточка сделки показывает реализации, но связь во внутренней таблице и обычным REST не отдаётся. **Дверь — скрытое семейство `crm.orderentity.*`** (нет в `methods`!):
- `crm.orderentity.list {filter:{orderId | ownerId, ownerTypeId:2}, select:['*']}` → `{orderId, ownerId, ownerTypeId}` — фильтруется И по заказу, И по сделке (проверено живьём);
- `crm.orderentity.add {fields:{orderId, ownerId, ownerTypeId:2}}` — ПИШЕТ привязку;
- `crm.orderentity.deleteByFilter {fields:{...}}` — удаляет.

## Стена 2 (СТОИТ): склад отгрузки через REST

- `sale.shipmentitemstore.*`, `sale.storebarcode.*`, `sale.basketitemreservation.*`, `crm.reservation.*` — `METHOD_NOT_FOUND` (перепробованы десятки имён, 2026-06-09/11).
- Запись склада в строку сделки: `crm.item.productrow.update` поле `storeId` **не принимает** (`INVALID_ARG_VALUE: Field 'storeId' not available for update`, 2026-06-11) — нативный механизм пишет его изнутри.
- Проводка `sale.shipment` без склада падает `DEDUCTION_STORE_ERROR`. **Вывод: «настоящую реализацию» с проводкой через REST собрать нельзя** — черновик можем, склад+«Провести» делает человек в нативной карточке (1 клик).
- **Чтение склада ЧЕРНОВИКА — МОЖНО** через резервы: `sale.order.get → basketItems[].reservations[].storeId` (когда менеджер выбрал склад в черновике, Битрикс вешает резерв; у резерва `quantity` может быть 0 — не смущаться, `storeId` валиден). После проведения резерв съедается — склад проведённой реализации не читается НИКАК. Проверено на мультискладе (заказ 722: склады 4/10/12/16 по строкам).

## Заказы (sale.order) и их побочки

- `sale.order.add` **авто-рождает дубль-сделку + контакт** на каждый заказ (CRM↔заказы сцеплены). `externalOrder:'Y'` НЕ спасает. Лечение: после создания читаем `crm.orderentity.list` по заказу, находим привязку к чужой свежей сделке (гард: `DATE_CREATE` < 15 мин) → `deleteByFilter` + `crm.deal.delete` + `crm.contact.delete`.
- Поле валюты — **`currency`, НЕ currencyId** (иначе ошибка). `lid:'s1'`, `personTypeId:6` (физлицо; 8 — юрлицо).
- **`userId` заказа = сотрудник-менеджер, не клиент** (проверено по нативным реализациям). Клиент в документе — свойства заказа: `sale.propertyvalue.list` (для personType 6: prop 40 «Имя Фамилия», 42 E-Mail, 44 Телефон, 52 Адрес). Запись — `sale.propertyvalue.modify` (метод существует; формат `{fields:{order:{id, propertyValues:[{orderPropsId, value}]}}}` живым тестом НЕ подтверждён).
- Нативная модель частичных отгрузок: **один заказ → много отгрузок** (`#N/2, /3, /4`); неотгруженный остаток Битрикс держит на системной отгрузке (`system:'Y'` — фильтровать в выборках!).

## Связь строк корзины со строками сделки: `crm_pr_`

`sale.order.get → basketItems[].xmlId === 'crm_pr_<N>'`, где N — id товарной строки CRM. Разворот: `crm.item.productrow.get {id:N}` → `{ownerType:'D', ownerId:<dealId>}`. Покрытие на портале фактически 100%. Внимание:
- ответ `.get` лежит в ключе **`productRow`** (не `item`);
- `crm.item.productrow.list` требует **`'=ownerType'`** в фильтре (именно с `=`);
- создавая корзину своими руками, проставляй `xmlId: crm_pr_<rowId>` — структура неотличима от нативной.

## Типы строк и каталога (баг «пропавший монитор», 2026-06-11)

- Строка сделки `TYPE`: **1 = товар, 4 = вариация (торговое предложение — ТОЖЕ товар!), 7 = работа/услуга.** Фильтруй «работа = 7, всё остальное = товар», иначе вариации невидимы при видимой сумме.
- Каталог: iblock **24** = товары (type 1 товар, 3 SKU-родитель, 7 услуга), iblock **26** = вариации (type 4, `parentId`). Остатки живут на вариациях. SKU-родители — контейнеры без остатков.
- `catalog.product.list` требует `iblockId` в select; цены — `catalog.price.list {catalogGroupId:2}` (BASE = розница); остатки — `catalog.storeproduct.list` (без фильтра amount — отдаёт и нули).

## Старый vs новый API товарных строк

`crm.deal.productrows.get` (старый): UPPER_CASE, есть STORE_ID/RESERVE_ID/RESERVE_QUANTITY/DATE_RESERVE_END. `crm.item.productrow.*` (новый): camelCase, есть `storeId` (read-only на запись!). У товаров проведённых реализаций `storeId` заполнен — его пишет нативный механизм.

## Фронтовый BX24 SDK флапает

`app.option.get`, `crm.deal.productrows.get` на свежесозданных сделках, инициализация при возврате из нативных окон — зависают. Лечение: серверный клиент для всего важного, `withTimeout`+`withRetry`+мягкие фолбэки для остального. Симптомы у юзера: «вкладка пустая после добавления», «таймаут 15с».

## OAuth локального приложения (мобильный режим)

- Б24 **игнорирует `redirect_uri`** — code приходит на зарегистрированный `/app/handler`; обмен на `https://oauth.bitrix.info/oauth/token/` (client_secret обязателен).
- `tok.domain` из обмена = `oauth.bitrix.info` — это НЕ REST-хост! REST слать на домен портала.
- Открытие портального приложения в мобильном вебе валит портал у всех; автономная страница (`/m`) — безопасна.

## Разное

- `entity.*` работает только в app-контексте (не вебхуком); `entity.add` — админская.
- `sale.shipment.list` иногда отдаёт сетевой `fetch failed` — лечится повтором (во все скрипты — ретраи).
- Идентификатор карточки реализации в UI = id отгрузки: `/shop/documents/details/sales_order/<shipmentId>/`. Сделка: `/crm/deal/details/<id>/`. Смарт-процесс: `/crm/type/<typeId>/details/<id>/`.
- Деплой нового бандла: первые секунды может отдавать 404 (прогрев) — повторить запрос.
