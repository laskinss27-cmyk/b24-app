# 🆘 SOS — что делать, когда что-то сломалось

Эта шпаргалка — чтобы поднять систему **самому**, по шагам. Команды можно копировать как есть.
Ничего не бойся: **данные ядра живут в Docker-томах и так просто не пропадают.** Почти любой сбой
лечится перезапуском. Если совсем тупик — внизу раздел «Ничего не помогает».

---

## 0. Куда вообще заходить

Вся «вычислялка» (ядро ERPNext + наше приложение + туннель + синк) живёт на **запасном ноуте (спейр)**.
Заходишь на него с рабочего ноута — в терминале (PowerShell или Git Bash):

```
ssh -i ~/.ssh/b24_homeserver rey@192.168.0.69
```

- `192.168.0.69` — адрес спейра в сети на работе. Если не подключается — **оба ноута должны быть в ОДНОМ роутере** (на работе их несколько). Адрес спейра мог смениться; посмотреть можно на самом спейре или спросить роутер.
- Зашёл — увидишь приглашение `rey@...:~$`. Это и есть «внутри сервера».

---

## 1. ПЕРВЫМ ДЕЛОМ ВСЕГДА — `status`

Зашёл на спейр → пиши:

```
status
```

Это покажет состояние ВСЕГО по-русски: ядро, приложение, дверь наружу, туннель, контейнеры, синк, диск/память.
- **Всё ✅ (зелёное)** — система здорова. Если жалуются на приложение — проблема НЕ здесь (скорее права доступа в Б24, см. ниже п.7).
- **Есть ⛔ (красное)** — смотри ниже нужный раздел.
- **ℹ️ жёлтое** — это не поломка, просто заметка (например, синк нагонит после включения).

> `status` — только чтение, ничего не ломает. Запускай сколько угодно.

### Что такое `status` и где он живёт

- Это **настоящая команда** на спейре: `/usr/local/bin/status` (доступна из любой папки, под любым входом). НЕ алиас — алиас однажды записался криво и ломался, поэтому сделали командой.
- Исходник — в репозитории: **`scripts/sos-status.sh`** (версионируется в git).
- Делает только чтение: дёргает ping ядра, health backend, публичную дверь, `systemctl is-active b24-tunnel`, `docker ps`, последний прогон синка (`~/sync/sync.log`), `df`/`free`. Токен ядра для счётчика товаров берёт из `~/sync/.env` (`ERPNEXT_TOKEN`).

### Если `status` вдруг «command not found» / показывает чушь

```bash
# Переустановить команду из репо-скрипта (с рабочего ноута):
scp -i ~/.ssh/b24_homeserver D:/Projects/b24-app/scripts/sos-status.sh rey@192.168.0.69:~/status.sh
ssh -i ~/.ssh/b24_homeserver rey@192.168.0.69 'sed -i "s/\r$//" ~/status.sh && sudo cp ~/status.sh /usr/local/bin/status && sudo chmod +x /usr/local/bin/status && status | head -3'
```
Если сам на консоли сервера и команда «висит» от старой сессии — `unalias status` или перелогиниться (`exit` → войти заново).

## 2. ⛔ «Контейнер упал» / приложение (backend) не отвечает

Симптом: в `status` ⛔ у Backend или у контейнеров.

```
docker restart b24-backend
```
Подожди 5 секунд, проверь:
```
status
```
Если backend всё ещё ⛔ — посмотри, на чём он спотыкается (последние строки лога):
```
docker logs --tail 40 b24-backend
```

---

## 3. ⛔ «Ядро ERPNext не отвечает» (команды в ядро не проходят)

Симптом: в `status` ⛔ у ядра, или приложение открывается, но склад/каталог пустые.

Перезапусти весь стек ядра (это безопасно, данные в томах сохраняются):
```
cd ~/erpnext && docker compose -f pwd.yml restart
```
Подожди ~30 секунд (ядро поднимается не мгновенно), потом:
```
status
```
Если ядро так и не встаёт — подними стек заново:
```
cd ~/erpnext && docker compose -f pwd.yml up -d
```

---

## 4. ⛔ «Туннель отвалился» / ⛔ «Дверь снаружи недоступна»

Симптом: backend и ядро ✅, но публичная дверь ⛔ (Битрикс снаружи не достаёт приложение).

```
sudo systemctl restart b24-tunnel
```
Проверь:
```
status
```
Посмотреть, что с туннелем:
```
systemctl status b24-tunnel --no-pager
```

---

## 5. ⛔ «Синк встал» (остатки в приложении устарели)

Сначала пойми, настоящий ли это сбой:
- Если ноут только что включили — синк сам нагонит в ближайшие :07 минут каждого часа. `status` это пишет жёлтым — тогда ничего не делай.
- Если ноут давно работает, а синк ⛔ красным — прогони вручную:
```
bash ~/sync/sync.sh
```
Потом глянь итог:
```
tail -n 5 ~/sync/sync.log
```
Должно закончиться «✅ СОШЛОСЬ В НОЛЬ».

## 6. 🔙 Откатить плохой деплой (после обновления приложения стало хуже)

Вернуть прошлый рабочий образ приложения:
```
docker rm -f b24-backend
docker run -d --name b24-backend --network erpnext_frappe_network -p 3000:8080 \
  --env-file ~/erpnext/backend.env --restart unless-stopped \
  sha256:a39f14bf54cbc896978bbf8c114113af551b466c394fc000ca7aa95baa2bca72
```
(Это образ «до доработок ремонтов 2026-06-17» — заведомо рабочий. Актуальные sha для отката Claude
держит в памяти `project_repairs`.) Проверь `status`.

## 7. Приложение «не найдено» у сотрудников, но `status` весь зелёный

Тогда дело **не в сервере**, а в **правах доступа к приложению в самом Битрикс24**
(в карточке приложения — кому разрешено пользоваться). Такое уже было 2026-06-17:
лечилось добавлением прав доступа в админке Б24. Сервер тут ни при чём.

---

## 8. После перезагрузки / включения ноута

Ничего вручную делать НЕ надо — всё поднимается само (Docker в автозапуске, контейнеры с
`restart: unless-stopped`, туннель в systemd, синк в cron). Просто зайди и проверь:
```
status
```
Через 1-2 минуты после старта всё должно быть ✅ (синк — жёлтым, нагонит в ближайший :07).

---

## 9. Базовые «глаза» вручную (если `status` почему-то не запустился)

```
docker ps                         # что сейчас запущено
docker ps -a                      # + что выключено
docker logs --tail 50 b24-backend # лог приложения
curl -s localhost:8080/api/method/ping   # ядро живо? → {"message":"pong"}
curl -s localhost:3000/health            # приложение живо? (тишина = ок, или JSON)
df -h /                           # сколько места на диске
free -h                           # сколько памяти
```

---

## 10. Ничего не помогает

- **Данные целы**, пока цел диск спейра — ядро и его БД лежат в Docker-томах (`erpnext_db-data`, `erpnext_sites`).
- Полный «перезапуск всего»: перезагрузи спейр (`sudo reboot`), подожди 2 минуты, зайди, `status`.
- Если и это не помогло — **не трогай дальше наугад**, опиши Claude, что показывает `status` и `docker ps -a`.
  Откат к рабочему состоянию почти всегда возможен — главное не паниковать и не удалять тома.

> ⚠️ Чего НЕ делать никогда: `docker compose down -v` (флаг `-v` сносит тома = данные), `docker volume rm`,
> `docker system prune --volumes`. Без `-v` — безопасно.
