# 🆘 SOS — что делать, когда что-то сломалось

Эта шпаргалка — чтобы поднять систему **самому**, по шагам. Команды можно копировать как есть.
Ничего не бойся: **данные ядра живут в Docker-томах и так просто не пропадают.** Почти любой сбой
лечится перезапуском. Если совсем тупик — внизу раздел «Ничего не помогает».

Развернуть систему с нуля (новый сервер) или восстановить данные из бэкапа — в [runbook.md](runbook.md).
Как всё устроено и связано — в [network.md](network.md).

---

## 0. Куда вообще заходить

Вся система (ядро ERPNext + наше приложение + дверь nginx + синк + бэкапы) живёт на **одном VPS**:

```
ssh root@201.51.12.57
```

- Вход — по ssh-ключу (заводится заранее, держится у ответственного). Пароль для входа не используем.
- Зашёл — увидишь приглашение `root@msk-1-vm-...:~#`. Это и есть «внутри сервера».
- Имя сервера наружу: `201.51.12.57.sslip.io` (это тот же IP, по нему приложение открывается в Б24).

---

## 1. ПЕРВЫМ ДЕЛОМ ВСЕГДА — `status`

Зашёл на VPS → пиши:

```
status
```

Покажет состояние ВСЕГО по-русски: ядро, приложение, дверь наружу, сертификат, контейнеры, синк, диск/память.
- **Всё ✅ (зелёное)** — система здорова. Если жалуются на приложение — проблема НЕ на сервере (скорее кеш
  Б24 или права доступа в Б24, см. п.7).
- **Есть ⛔ (красное)** — смотри ниже нужный раздел.
- **ℹ️ жёлтое** — не поломка, заметка (например, синк нагонит в ближайший :07).

> `status` — только чтение, ничего не ломает. Запускай сколько угодно.

### Если `status` вдруг «command not found»

```bash
# Переустановить команду из репо-скрипта (с машины, где склонирован репозиторий):
scp scripts/sos-status.sh root@201.51.12.57:~/status.sh
ssh root@201.51.12.57 'sed -i "s/\r$//" ~/status.sh && cp ~/status.sh /usr/local/bin/status && chmod +x /usr/local/bin/status && status | head -3'
```

## 2. ⛔ «Контейнер упал» / приложение (backend) не отвечает

Симптом: в `status` ⛔ у Backend или у контейнеров.

```
docker restart b24-backend
```
Подожди 5 секунд, проверь `status`. Если backend всё ещё ⛔ — посмотри, на чём спотыкается:
```
docker logs --tail 40 b24-backend
```

---

## 3. ⛔ «Ядро ERPNext не отвечает» (склад/каталог пустые)

Симптом: в `status` ⛔ у ядра, или приложение открывается, но склад/каталог пустые.

Перезапусти стек ядра (безопасно, данные в томах сохраняются):
```
cd /root/erpnext && docker compose -p erpnext -f pwd.yml restart
```
Подожди ~30 секунд (ядро поднимается не мгновенно), потом `status`.
Если так и не встаёт — подними стек заново:
```
cd /root/erpnext && docker compose -p erpnext -f pwd.yml up -d
```

---

## 4. ⛔ «Дверь снаружи недоступна» (Битрикс не открывает приложение)

Симптом: backend и ядро ✅, но снаружи приложение не грузится.

Проверь дверь с любой машины:
```
curl -I https://201.51.12.57.sslip.io/health
```
- **Не отвечает / ошибка nginx** → на VPS:
  ```
  systemctl status nginx --no-pager
  systemctl restart nginx
  tail -30 /var/log/nginx/error.log
  ```
- **Ошибка сертификата / просрочен** → проверь и обнови:
  ```
  certbot certificates
  certbot renew --force-renewal && systemctl reload nginx
  ```
  (Обычно продление автоматическое; ручное нужно только если таймер не сработал.)

---

## 5. ⛔ «Синк встал» (остатки в приложении устарели)

Сначала пойми, настоящий ли сбой:
- Синк идёт каждый час в :07. `status` это пишет — если последний прогон свежий, всё ок.
- Если давно не было / ⛔ красным — прогони вручную:
```
bash /root/sync/sync.sh
tail -n 8 /root/sync/sync.log
```
Должно закончиться «расхождений 0».

---

## 6. 🔙 Откатить плохой деплой (после обновления приложения стало хуже)

Backend собирается из исходников. Откат = вернуться на прошлый рабочий коммит и пересобрать.
Подробные шаги (сборка/запуск) — в [runbook.md](runbook.md) раздел «Деплой / откат backend». Кратко:
```
# на машине с репозиторием: переключиться на заведомо рабочий коммит/тег, собрать образ,
# залить на VPS и перезапустить b24-backend. См. runbook.
```
Если образ предыдущей версии ещё лежит в docker на VPS (`docker images | grep b24-app`) — можно быстро
вернуть контейнер на него (точную команду запуска см. в runbook).

---

## 7. Приложение «не открывается» / «Ошибка при показе», но `status` зелёный

Сервер тут ни при чём. Два частых случая:

- **«Ошибка при показе приложения»** сразу после смены домена/настроек приложения → это **кеш Б24**.
  Лечится жёстким обновлением страницы портала: **Ctrl+Shift+R**. (Сервер при этом отдаёт всё корректно —
  можно проверить `curl -I https://201.51.12.57.sslip.io/health` → 200.)
- **«Приложение не найдено» у части сотрудников** → дело в **правах доступа к приложению в самом Б24**
  (в карточке приложения — кому разрешено). Лечится в админке Б24, не на сервере.

---

## 8. После перезагрузки VPS

Ничего вручную делать НЕ надо — всё поднимается само (Docker в автозапуске, контейнеры с
`restart: unless-stopped`, nginx и certbot-таймер в systemd, синк/бэкап в cron). Просто зайди и проверь:
```
status
```
Через 1-2 минуты после старта всё должно быть ✅ (синк — нагонит в ближайший :07).

## 9. Базовые «глаза» вручную (если `status` не запустился)

```
docker ps                              # что сейчас запущено
docker ps -a                           # + что выключено
docker logs --tail 50 b24-backend      # лог приложения
curl -s localhost:8080/api/method/ping # ядро живо? → {"message":"pong"}
curl -s localhost:3000/health          # backend живо? → JSON {"ok":true,...}
curl -I https://201.51.12.57.sslip.io/health  # дверь снаружи → 200
df -h /                                # сколько места на диске
free -h                                # сколько памяти
```

## 10. Восстановить данные ядра из бэкапа

Если данные ядра повреждены/потеряны (но VPS жив) — восстановление из последнего дампа.
Бэкапы: локально `/root/core-backups/` и на Б24-Диске. **Полная пошаговая инструкция — в [runbook.md](runbook.md)
раздел «Восстановление из бэкапа».** Суть:
```
# скопировать дамп в контейнер ядра и:
docker exec erpnext-backend-1 bench --site frontend restore <db.sql.gz> \
  [--with-public-files <files.tar>] [--with-private-files <private-files.tar>]
```
> ⚠️ `restore` ПЕРЕЗАПИШЕТ текущую БД ядра. Делай осознанно, сверь дату дампа.

## 11. Ничего не помогает

- **Данные целы**, пока цел диск VPS — ядро и его БД лежат в Docker-томах (`erpnext_db-data`, `erpnext_sites`).
- Полный «перезапуск всего»: `reboot`, подожди 2 минуты, зайди, `status`.
- **Если VPS потерян целиком** (провайдер, диск, снос) — система разворачивается заново на новой машине
  из **GitHub + бэкапа**, без чьих-либо личных компьютеров. См. [runbook.md](runbook.md) «Развернуть с нуля».
- Если непонятно — **не трогай дальше наугад**, зафиксируй что показывает `status` и `docker ps -a`.

> ⚠️ Чего НЕ делать НИКОГДА: `docker compose down -v` (флаг `-v` сносит тома = данные), `docker volume rm`,
> `docker system prune --volumes`. Без `-v` — безопасно.
