# Статус приложения в нативной карточке сделки Bitrix24

Заметка для будущей реализации. На момент исследования никаких полей, роботов и настроек карточки в Bitrix24 не создавали и не изменяли.

## Задача

Выводить рассчитанное приложением состояние сделки прямо в нативной вкладке сделки «Общее», чтобы:

- менеджер видел понятный текущий статус;
- робот Bitrix24 мог читать статус и выбирать дальнейшее действие;
- статус не зависел от открытия нашей вкладки приложения.

## Найденное решение

Использовать стандартное пользовательское поле сделки типа `enumeration` («Список»), а не отдельную встройку.

Предлагаемое рабочее название: **«Статус комплектации»** или **«Статус Умного дома»**.

Поле:

- хранится непосредственно в сделке Bitrix24;
- отображается в нативной карточке на вкладке «Общее»;
- обновляется приложением через `crm.deal.update`;
- доступно в условиях роботов;
- может отслеживаться триггером Bitrix24 «Отследить изменение полей».

Тип «Список» предпочтительнее текстовой строки: робот получает фиксированный набор значений без опечаток и разных формулировок.

## Предполагаемая схема

1. Приложение определяет состояние сделки после значимого действия.
2. Приложение записывает значение в пользовательское поле через `crm.deal.update`.
3. Поле отображается менеджеру в нативной вкладке «Общее».
4. Триггер «Отследить изменение полей» реагирует на изменение статуса.
5. Триггер переводит сделку на нужную стадию либо запускает предусмотренную автоматизацию.
6. Роботы используют значение поля в условиях `равно`, `не равно`, `заполнено` и других.

Важно: одно изменение поля само по себе не гарантирует повторный запуск уже прошедших роботов стадии. Для немедленной реакции следует использовать триггер изменения поля либо заранее спроектированную схему стадий и ожидающих роботов.

## Возможные значения

Это только черновой список. Перед реализацией его необходимо согласовать с фактической логикой роботов:

- Расчёт КП
- Ожидается выбор клиента
- КП выбрано
- Требуется снабжение
- Комплектация
- Готово к монтажу
- Монтаж
- Завершено
- Требует внимания

Нужно отдельно решить, достаточно ли одного линейного статуса. Некоторые состояния могут существовать одновременно, например снабжение и подготовка монтажа. В таком случае надёжнее разделить их на несколько полей или явно определить приоритет состояний.

## Размещение в нативной карточке

Bitrix24 позволяет управлять разделами и полями нативной карточки через:

- `crm.item.details.configuration.get`;
- `crm.item.details.configuration.set`;
- `crm.item.details.configuration.forceCommonScopeForAll`.

Для сделки используется `entityTypeId: 2`. Настройки могут различаться по воронкам через `extras.dealCategoryId`.

Поле можно добавить в верхний раздел общего вида карточки и установить `optionFlags: 1`, чтобы оно показывалось всегда.

Осторожно: метод `crm.item.details.configuration.set` передаёт конфигурацию карточки целиком. При автоматизации необходимо сначала прочитать действующую конфигурацию через `get`, аккуратно добавить поле и только затем сохранить результат. Нельзя заменять карточку заранее составленным неполным массивом.

На первом этапе безопаснее создать поле и добавить его в общий вид карточки вручную через интерфейс администратора Bitrix24. Программное управление карточкой можно добавить после проверки на одной воронке.

## Уже существующие похожие поля

В сохранённом снимке полей портала найдены:

- `UF_CRM_1756888012167` — строковое поле «Статус»;
- `UF_CRM_1760584102770` — строковое поле «Статус»;
- `UF_CRM_6878FF1CC2E7C` — строковое поле «Статус (из базы)».

В коде приложения эти поля не используются. Однако они могут быть задействованы в старых роботах или других интеграциях, поэтому переиспользовать их без отдельной проверки нельзя.

Рекомендуемый вариант — отдельное поле-список с однозначным названием и стабильным внешним кодом (`XML_ID`).

## Что не рекомендуется

### Отдельная вкладка или placement

`CRM_DEAL_DETAIL_TAB` создаёт отдельную вкладку, а не выводит данные в «Общее». Роботы не используют содержимое iframe как поле сделки.

### Пользовательский тип поля с виджетом

Bitrix24 позволяет встроить интерфейс приложения внутрь пользовательского поля. Это подходит для кнопок и динамического интерфейса, но для машинного статуса избыточно:

- появляется зависимость от загрузки обработчика приложения;
- сложнее настройка и сопровождение;
- роботам всё равно требуется простое сохранённое значение.

Для статуса следует использовать стандартный тип `enumeration`.

## Перед будущей реализацией

1. Согласовать назначение статуса: только склад/комплектация или общее состояние сделки.
2. Согласовать окончательные значения и переходы между ними.
3. Проверить существующие три поля «Статус» и текущие роботы портала.
4. Решить, должен ли статус быть одним полем или несколькими независимыми признаками.
5. Создать поле на тестовой воронке и вручную добавить его в общий вид карточки.
6. Настроить тестовый триггер изменения поля.
7. Только после проверки подключать автоматическое вычисление статуса в приложении.
