## B. Снабжение (отдельный интерфейс снабженца)

- Менеджер создаёт заказ (из вкладки, см. A) → падает снабженцу.
- **Снабженец**: левое меню — Остатки по складам, Заказы (фильтр активные/неактивные), Логистика, Закупки, Документы склада.
- К каждой заявке — **дерево**: все позиции → снабженец расставляет источник для каждой (с Парнаса / с Девяткина / закупить) → сохраняет → создаются дочерние **логистические** и **закупочные** воронки.
- 🔑 **Менеджер НИКОГДА не выбирает склад-источник** — это только снабженец (видит всю картину, оптимизирует логистику, знает где что лежит). У менеджера такой возможности в интерфейсе быть не должно.
- Одинаковые позиции из разных заявок **НЕ складывать** (нужно понимать что откуда; разное состояние: новый запакованный vs распакованный со стенда).
- Менеджер может создать перемещение сам (не под КП, себе на точку) — только запрашивает что хочет + примечание, без склада-источника; появляется у снабженца без привязки к сделке.

## C. Перемещения (честный транзит)

- **Склад транзита**: при отгрузке товар уходит на транзит (не у отправителя, не у получателя), пока получатель не примет. Одно перемещение = маршрут в 3 точки (откуда → транзит → куда).
- Приём на ТТ: менеджер в складском учёте видит перемещения на его точку → прощёлкивает галочки что пришло → подтверждает. Лишнее/недостача подсвечивается **красным**.
- Отгрузка: менеджер галочками отмечает что отдал → водителю → инфо снабженцу.
- **Кто что жмёт** (чтобы снабженец не кликал 300 раз): отгружает и принимает на точках **менеджер** (списание в транзит + приём), а **снабженец ЗАВЕРШАЕТ цепочку** — проводит перемещение из черновика, когда убедился что всё пришло.
- Уведомления в чат Б24 (как задача): «отправь / прими перемещение» + ссылка.
- По сути упрощённая логика 1С.

## D. Приходы / закупки / списания / отчёты

- Менеджер может делать **черновики** приходов/списаний для снабженца (закупил сам на зоне; брак) — снабженец подтверждает. Списания согласуются.
