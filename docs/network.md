# 🌐 Сетевая схема — как Битрикс достаёт приложение, а приложение — ядро

Документ «чтобы разобраться без Claude». Если что-то лежит — смотри раздел «Если упало: куда смотреть».
Аварийные команды (рестарты) — в [SOS.md](SOS.md). Здесь — КАРТА: что где живёт и как связано.

---

## Картина одним абзацем

Менеджер открывает наше приложение в Битриксе → браузер грузит наш UI по **HTTPS с VPS reg.ru** →
VPS (nginx) проксирует запрос через **обратный ssh-туннель** на **спейр-ноут** (он на работе, в LAN) →
там **backend** отдаёт фронт-бандл и сам ходит: в **ядро ERPNext** по localhost (соседний контейнер) и в
**Битрикс REST** наружу. Битрикс НЕ ходит в наше ядро напрямую — только открывает приложение.

```
                    ИНТЕРНЕТ
  менеджер ─(открывает приложение в Б24, браузер)─┐
                                                  ▼
        ┌──────────────────────────────────────────────────┐
        │  VPS reg.ru  194.226.97.154 (белый IP)            │
        │  nginx :443  (валидный Let's Encrypt сертификат)  │
        │  имя: 194-226-97-154.regru.cloud                  │
        │  proxy_pass → 127.0.0.1:3001 ──────────┐          │
        └────────────────────────────────────────┼─────────┘
                                                  │  обратный ssh-туннель
                                                  │  (спейр САМ держит соединение наружу)
                                                  ▼
        ┌──────────────────────────────────────────────────┐
        │  СПЕЙР-ноут (на работе, LAN 192.168.0.69)         │
        │  ┌────────────────────┐   ┌────────────────────┐  │
        │  │ backend (:3000)    │──▶│ ядро ERPNext       │  │
        │  │ b24-backend        │   │ frontend:8080      │  │
        │  │ — отдаёт фронт     │   │ (одна docker-сеть, │  │
        │  │ — /api/*           │   │  localhost)        │  │
        │  └─────────┬──────────┘   └────────────────────┘  │
        └────────────┼─────────────────────────────────────┘
                     │ исходящие REST-запросы
                     ▼
              Битрикс24 (облако, umniydom.bitrix24.ru)
```

**Ключевая мысль:** наружу торчит ТОЛЬКО VPS (белый IP). Спейр сам тянется к VPS изнутри LAN — поэтому
**динамический IP / NAT ноута неважны** (нам не нужен входящий проброс, ноут инициирует соединение сам).

## Узел 1 — VPS reg.ru (публичная дверь)

- IP `194.226.97.154`, имя `194-226-97-154.regru.cloud` (reg.ru даёт бесплатно, резолвится).
- **nginx**, конфиг нашего блока: `/etc/nginx/conf.d/b24-app.conf`
  - `listen 192.168.0.194:443 ssl` (192.168.0.194 — внутренний адрес самого VPS; снаружи приходит на 194.226.97.154:443).
  - `server_name 194-226-97-154.regru.cloud` — по SNI наш блок выигрывает у панельных vhost'ов ISPmanager.
  - `location / { proxy_pass http://127.0.0.1:3001; }` — на туннель.
  - SSL: Let's Encrypt, `/etc/letsencrypt/live/194-226-97-154.regru.cloud/` (certbot, авто-продление).
- На VPS ТАКЖЕ живёт панель reg.ru (ISPmanager) на других именах — наш `conf.d/` отдельный, не конфликтует.
- Заход (управление): `ssh -i ~/.ssh/b24_bridge root@194.226.97.154`.
- ⚠️ Если ISPmanager когда-нибудь вычистит `conf.d/` — наш блок восстановить (есть в истории/доках).

## Узел 2 — обратный ssh-туннель (связь VPS ↔ спейр)

- Живёт **на спейре** как systemd-сервис **`b24-tunnel`** (`Restart=always`, автозапуск).
- Команда: `ssh -i ~/.ssh/vps_tunnel -N -R 127.0.0.1:3001:localhost:3000 root@194.226.97.154`
  - читается так: «прокинь порт VPS `127.0.0.1:3001` на мой локальный `:3000` (backend)».
- Ключ туннеля: `~/.ssh/vps_tunnel` на спейре (его публичная часть — в `authorized_keys` на VPS).
- Проверка: на спейре `systemctl is-active b24-tunnel` → `active`. Рестарт: `sudo systemctl restart b24-tunnel`.
- Это НЕ ngrok/cloudflared (cloudflared в РФ мёртв — SNI-блок, проверено) и НЕ VPN. Обычный обратный ssh.

## Узел 3 — спейр (вся вычислялка)

- На работе, LAN `192.168.0.69`, пользователь `rey`, заход: `ssh -i ~/.ssh/b24_homeserver rey@192.168.0.69`.
- Docker-сеть `erpnext_frappe_network`, в ней рядом:
  - **`b24-backend`** — наш Fastify-контейнер, порт `3000:8080`. Раздаёт фронт-бандл (`@fastify/static`) + `/api/*`. env в `~/erpnext/backend.env` (в т.ч. `ERPNEXT_URL=http://frontend:8080`).
  - **ядро ERPNext** — ~10 контейнеров (`frontend`, `backend`, `db`, redis, воркеры). backend ходит в него по `http://frontend:8080` (имя сервиса в docker-сети = localhost-скорость, без туннелей).
- Деплой/откат backend — [runbook.md](runbook.md). Состояние всего — команда `status` ([SOS.md](SOS.md)).

## Куда что ходит (важно для понимания)

- **Входящие** (снаружи): только на VPS:443 → туннель → backend:3000. Больше ниоткуда наружу не торчим.
- **Backend → ядро**: исходящие по docker-сети (`frontend:8080`), внутри спейра.
- **Backend → Битрикс REST**: исходящие к `umniydom.bitrix24.ru` (читаем/пишем Б24 токеном юзера). Битрикс сам в нас REST не шлёт — он лишь открывает приложение (iframe).
- **Синк остатков**: cron на спейре (`~/sync`), тоже исходящие к Б24 + локально в ядро.

## Если упало: куда смотреть (по слоям, сверху вниз)

1. **`status` на спейре** — первым делом. Он сразу покажет, какой слой красный.
2. **Приложение не открывается у всех** → проверь по цепочке:
   - публичная дверь: `curl https://194-226-97-154.regru.cloud/health` (с любой машины). Не 200 → проблема на VPS/туннеле.
   - туннель: на спейре `systemctl is-active b24-tunnel`. Не active → `sudo systemctl restart b24-tunnel`.
   - backend: `status` / `docker restart b24-backend`.
   - VPS жив? nginx работает? (логи nginx выше).
3. **Открывается, но склад пустой** → ядро (`status` → ядро, рестарт стека — см. SOS.md).
4. **«Приложение не найдено» у части юзеров, а `status` зелёный** → это НЕ сеть, а **права доступа к приложению в Б24** (см. SOS.md, п.7).
5. **Спейр выключен / нет сети** → 502 у всех, пока не включится (данные целы, поднимется сам).

Контроль здоровья снаружи: на VPS есть heartbeat (`/root/b24-heartbeat.sh`, cron каждые 5 мин) — стучит в ядро через туннель и шлёт Сергею Б24-уведомление при падении (ловит даже когда спейр выключен).

## ⚠️ Известные пробелы (TODO на боевой сервер, не на канарейку)

Честный список, чтобы не забыть при переезде на сервер компании:
- **Метрик/алертов мало.** Есть `responseTime` в логах + heartbeat, но нет агрегации (p95-латентность, error-rate, счётчик ретраев) — а это понадобится, чтобы доказать стабильность перед раскаткой на всех 25 менеджеров.
- **Нет сборщика фронт-ошибок** (Sentry и т.п.). Юзеру текст ошибки показываем, но централизованно фронт-ошибки не собираем.
- **Статика идёт через спейр.** Спейр спит → менеджеры видят 502/таймаут, а не аккуратное «приложение недоступно». На бою можно отдавать статику-заглушку с VPS, чтобы сообщение показывалось даже при спящем ноуте.
- **Аптайм = аптайм ноута** (вкл + интернет + туннель). Для 4 канареек ок, для 25 боевых — нормальный сервер компании (туда же RAID, нормальный HTTPS без туннеля, бэкапы).
