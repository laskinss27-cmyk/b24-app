## Картина одним абзацем

Менеджер открывает наше приложение в Битриксе → браузер грузит наш UI по **HTTPS прямо с VPS**
(`https://201.51.12.57.sslip.io`) → **nginx** на VPS проксирует на локальный **backend** (`127.0.0.1:3000`) →
backend отдаёт фронт-бандл и сам ходит: в **ядро ERPNext** по docker-сети (`http://frontend:8080`) и в
**Битрикс REST** наружу. Битрикс НЕ ходит в наше ядро напрямую — только открывает приложение в iframe.

```
                    ИНТЕРНЕТ
  менеджер ─(открывает приложение в Б24, браузер)─┐
                                                  ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  VPS  201.51.12.57  (белый IP, Ubuntu 22.04, 24/7)            │
   │  имя: 201.51.12.57.sslip.io  (резолвится в этот же IP)         │
   │                                                                │
   │  nginx :443 ssl  (Let's Encrypt, авто-продление)               │
   │       │  proxy_pass → 127.0.0.1:3000                           │
   │       ▼                                                        │
   │  ┌────────────────────┐   ┌─────────────────────────────────┐ │
   │  │ b24-backend        │──▶│ ядро ERPNext (~10 контейнеров)   │ │
   │  │ 127.0.0.1:3000     │   │ frontend:8080 (одна docker-сеть) │ │
   │  │ — отдаёт фронт     │   │ db / redis / воркеры             │ │
   │  │ — /api/*           │   │ данные в docker-томах            │ │
   │  └─────────┬──────────┘   └─────────────────────────────────┘ │
   │            │  синк + бэкап (cron на этом же VPS)               │
   └────────────┼───────────────────────────────────────────────── ┘
                │ исходящие REST-запросы
                ▼
         Битрикс24 (облако, umniydom.bitrix24.ru)
```

**Ключевая мысль:** всё на одной публичной машине, без туннелей. Наружу торчит **только nginx:443**
(и ssh:22). Backend и ядро слушают `127.0.0.1` — снаружи невидимы, ходить в них можно только через nginx
или с самого VPS. Машина в дата-центре, работает 24/7, переживает перезагрузку сама.

## Узел 1 — nginx (публичная дверь)

- IP `201.51.12.57`, имя **`201.51.12.57.sslip.io`** (сервис sslip.io резолвит имя в сам IP — свой домен не нужен).
- **nginx**, конфиг: `/etc/nginx/sites-available/b24` (симлинк в `sites-enabled/`).
  - `listen 443 ssl` + `listen 80` (80 редиректит на 443 и используется certbot для продления).
  - `server_name 201.51.12.57.sslip.io`.
  - `location / { proxy_pass http://127.0.0.1:3000; }` — на локальный backend.
  - SSL: Let's Encrypt, `/etc/letsencrypt/live/201.51.12.57.sslip.io/`, **авто-продление** (systemd-таймер certbot).
- Фаервол `ufw`: наружу открыты только **22 (ssh), 80, 443**. Остальное закрыто.

## Узел 2 — backend (наше приложение)

- Контейнер **`b24-backend`**, порт **`127.0.0.1:3000`** (наружу не публикуется — только через nginx).
- Образ **пересобирается из исходников**: в репозитории есть `Dockerfile` (см. [runbook.md](runbook.md) — сборка и запуск).
- env: `/root/erpnext/backend.env` (в т.ч. `ERPNEXT_URL=http://frontend:8080`, `PUBLIC_BASE_URL=https://201.51.12.57.sslip.io`).
- В docker-сети `erpnext_frappe_network` — рядом с ядром, ходит в него по `http://frontend:8080`.
- Раздаёт фронт-бандл (`@fastify/static`) + `/api/*` + placement-обработчики Б24.

## Узел 3 — ядро ERPNext

- ~10 контейнеров (`frontend`, `backend`, `db` mariadb, redis ×2, воркеры, scheduler, websocket).
- Compose: `/root/erpnext/pwd.yml`. Проект docker — `erpnext`. Сеть — `erpnext_frappe_network`.
- `frontend` опубликован только на `127.0.0.1:8080` (наружу закрыт; backend ходит по docker-сети).
- **Данные — в docker-томах** `erpnext_db-data` (БД) и `erpnext_sites` (site_config с ключом шифрования + фото).
  Тома при `restart`/`down` (без `-v`) НЕ теряются.
- Метка `env_label = PROD-COMPANY-VPS` в site_config (чтобы случайно не спутать боевое ядро с тестовым).

## Узел 4 — синк и бэкап (cron на VPS)

- `/root/sync/` — скрипты на Node (tsx). cron:
  - `7 * * * *` — `sync.sh` (ежечасно: каталог + остатки Б24 → ядро, flock + watchdog).
  - `0 12 * * *` — `core-backup.sh` (ежедневно бэкап БД; раз в неделю + файлы — см. [SOS.md](SOS.md)).
- Синк ходит в ядро по `http://localhost:8080` и в Б24 REST. `.env` в `/root/sync/.env`.

## Куда что ходит (важно для понимания)

- **Входящие** (снаружи): только nginx:443 → backend:3000. Больше ниоткуда наружу не торчим.
- **Backend → ядро**: исходящие по docker-сети (`frontend:8080`), внутри VPS.
- **Backend → Битрикс REST**: исходящие к `umniydom.bitrix24.ru` (читаем/пишем Б24 токеном юзера).
  Битрикс сам в нас REST не шлёт — он лишь открывает приложение (iframe).
- **Синк/бэкап**: cron на VPS, исходящие к Б24 + локально в ядро + бэкап БД на Б24-Диск.

## Где живёт «источник правды» (без этого не восстановиться)

| Что | Где | Зачем |
|---|---|---|
| Код приложения + инфра-скрипты + `Dockerfile` + `pwd.yml` | **GitHub** `github.com/laskinss27-cmyk/b24-app` | пересобрать образ и поднять стек с нуля |
| Дамп БД ядра (все документы и проводки) | локально `/root/core-backups` **+ Б24-Диск** | восстановить данные ядра |
| Файлы ядра (фото/вложения) | локально `/root/core-backups` (раз в неделю) | восстановить вложения; фото также восстановимы из Б24 синком |
| Секреты (`backend.env`, `/root/sync/.env`, токены) | только на VPS | НЕ в git; при потере VPS — пересоздать (см. runbook) |
| Доступ к VPS (ssh-ключ) | у владельца/админа | без ключа на VPS не зайти — завести свой заранее |

> Принцип: всё, что нужно для восстановления, живёт либо в **GitHub** (код/конфиги), либо в **бэкапе**
> (данные на VPS + Б24-Диск). Личные машины разработчиков в схему НЕ входят — на них не закладываться.

## Где логи (если что-то падает)

Сначала зайди на VPS: `ssh root@201.51.12.57` (по своему ключу).

| Что | Команда | Что видно |
|---|---|---|
| Состояние всего | `status` | ядро / backend / дверь / сертификат / контейнеры / синк / диск разом |
| Backend | `docker logs --tail 50 b24-backend` | входящие запросы (reqId, responseTime), ошибки роутов |
| Ядро | `docker logs --tail 50 erpnext-backend-1` | ошибки ERPNext |
| nginx | `tail -50 /var/log/nginx/error.log /var/log/nginx/access.log` | дошёл ли запрос браузера до VPS, ошибки прокси |
| Сертификат | `certbot certificates` | срок действия, авто-продление |
| Синк | `tail -50 /root/sync/sync.log` | прогоны синка, «расхождений 0» |
| Бэкап | `tail -20 /root/sync/core-backup.log` | последний бэкап, ушёл ли на Б24-Диск |

## Если упало: куда смотреть (по слоям, сверху вниз)

1. **`status` на VPS** — первым делом. Сразу покажет, какой слой красный.
2. **Приложение не открывается у всех** → проверь по цепочке:
   - дверь снаружи: `curl https://201.51.12.57.sslip.io/health` (с любой машины). Не 200 → проблема на VPS.
   - nginx: `systemctl status nginx` + лог error.log. Рестарт: `systemctl restart nginx`.
   - сертификат не протух? `certbot certificates`.
   - backend: `status` / `docker restart b24-backend`.
3. **Открывается, но склад/каталог пустой** → ядро (`status` → ядро, рестарт стека — см. SOS.md).
4. **«Ошибка при показе приложения» в Б24, а сервер зелёный** → почти всегда **кеш Б24** после смены
   домена/настроек: **Ctrl+Shift+R** на портале. Если не помогло — права доступа к приложению в Б24 (см. SOS.md).
5. **VPS целиком недоступен** (ssh не идёт, дверь молчит) → проблема у провайдера VPS. Если машина потеряна —
   развернуть заново из GitHub + восстановить данные из бэкапа (см. [runbook.md](runbook.md) «Развернуть с нуля»).

## ⚠️ Известные пробелы / на будущее

- **sslip.io — внешняя зависимость** (только для резолва имени). Надёжнее свой домен: A-запись на `201.51.12.57`
  + новый сертификат + смена `PUBLIC_BASE_URL` и URL в карточке Б24 (~10 минут). Пока работает sslip.io.
- **Бэкап рядом с ядром.** Дамп БД дублируется на Б24-Диск (защита от гибели VPS), файлы (фото) офсайта
  не имеют — они восстановимы из Б24 синком. Полноценный внешний офсайт — на будущее.
- **Метрик/алертов мало** — есть `responseTime` в логах, но нет агрегации (p95, error-rate) и сборщика фронт-ошибок.
- **Пароли по умолчанию** — root VPS и admin/admin ядра стоит сменить (ядро за фаерволом, но всё же).
