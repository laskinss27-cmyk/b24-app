# Б24-приложение — обзор для технаря

Внутренняя система складского и торгового учёта поверх Битрикс24, со своим складским «ядром» и полностью **self-hosted** инфраструктурой.

## Зачем
Б24 как CRM есть, но складской учёт в нём слабый: нет нормальных остатков по складам, перемещений, инвентаризации, привязки «сделка → товар → склад». Задача — закрыть это, не уводя людей из привычного Битрикса.

## Архитектура
```
Битрикс24 (облако) ──HTTPS──> корп-VPS 201.51.12.57 (Ubuntu, 24/7)
                              ┌──────────────────────────────────────┐
                              │ nginx :443 (Let's Encrypt, sslip.io)  │
                              │   → backend 127.0.0.1:3000            │
                              │ Docker-сеть erpnext_frappe_network:   │
                              │  • backend (Node/TS/Fastify)          │
                              │  • ERPNext/Frappe (складское ядро)    │
                              │     MariaDB + Redis + воркеры         │
                              │ + синк и бэкапы (cron)                │
                              └──────────────────────────────────────┘
```
- Приложение встраивается в Б24 как iframe-вкладки (placement) — backend генерит их и раздаёт собранный фронт.
- Публичная «дверь» — nginx с TLS прямо на VPS (`201.51.12.57.sslip.io`), без туннелей. С Yandex Cloud съехали, своя инфра.
- backend и складское ядро — в одной docker-сети (обмен по `frontend:8080`); наружу торчат только nginx:443 и ssh.

## Стек
- **Frontend:** React + Vite + TypeScript, монорепо (`packages/frontend|backend|shared`).
- **Backend:** Node + TypeScript + Fastify; REST-интеграция с Б24 (вебхуки/placement), entity-store Б24 для своих документов.
- **Складское ядро:** ERPNext (Frappe, MariaDB, Redis) — каталог, остатки по складам, проводки, сверки.
- **Инфра:** один VPS, Docker, nginx + Let's Encrypt, cron-синк и бэкапы (БД на Б24-Диск), команда `status` + рунбук восстановления из GitHub+бэкапа (`docs/runbook.md`, `docs/SOS.md`, `docs/network.md`).

## Масштаб данных
- Мигрирован весь каталог Б24 в ядро: **~5000 SKU** (товары + цветовые/поставщицкие вариации, наследование бренд/модель/раздел).
- **9 складов**, ежечасный авто-синк остатков, сверка «в ноль» (расхождений 0).
- Остатки в приложении берутся **из ядра** (мягкий фолбэк на Б24).

## Модули
- **Вкладка товаров сделки** — остатки по складам из ядра, статусы позиций (хватает / надо переместить / нет), реализация с группировкой по складам (Delivery Note).
- **Инвентаризация** — десктоп + мобильный обход по QR (телефон сканирует точку → подсчёт склада), черновики/отправка, акт разногласий, документ в ядре.
- **Ремонты (RMA)** — приём оборудования, статусы по движению, документы на Диск Б24, своя нумерация, права на цену.
- **Каталог-браузер, отчёт по продажам, генератор КП** (перенос в Б24).
- **В работе:** полный складской учёт — перемещения (честный транзит через transit-склад), списания, оприходования, реализации в едином окне.

## Инженерные детали, которыми не стыдно
- Идемпотентная миграция/синк (повторный прогон догоняет дельту, не дублирует).
- Сверка остатков с допуском на машинный эпсилон; пагинация Б24 по курсору `next` (а не по размеру страницы — иначе вечный цикл на кратных 50).
- Жёсткие таймауты фаз синка с убийством всей process-group (без осиротевших процессов).
- Проверенный на проде механизм складских проводок: Material Receipt / Transfer / Issue через REST ядра с проведением и откатом.

> Личные/рабочие заметки ассистента в обзор не входят — это техническая картина системы.
